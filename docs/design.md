# Bilibili 音乐播放器 (开源项目) 技术设计文档

## 1. 项目概述

本项目旨在开发一款现代化的跨平台桌面音乐播放器。核心特点是允许用户通过 B站 的 URL 或 BV 号导入视频音频，将 B站 作为一个庞大的音乐曲库。软件提供类似主流音乐软件的管理体验（歌单、自定义元数据、本地缓存），并支持 B站 账号登录以获取高音质音频。

## 2. 核心技术栈 (基于 Flutter)

- **前端框架 (UI & 跨平台):** **Flutter** (Dart)。支持 Windows, macOS, Linux 一键编译。
- **状态管理:** **Riverpod** 或 **Bloc**。推荐 Riverpod，类型安全且非常适合处理复杂的异步数据（如网络请求和播放器状态）。
- **媒体播放内核:** **media_kit**。这是一个基于 `libmpv` 的 Flutter 跨平台音视频库，性能极强，完美支持 B站的 dash/m4s 流播放，且自带硬件加速。
- **本地数据库:** **Isar Database**。一个专为 Flutter 打造的超快速 NoSQL 数据库，非常适合存储包含大量元数据的离线歌单、歌曲信息和用户自定义修改。
- **网络与 API 解析:** **Dio** (HTTP 客户端) + 独立抽离的 `Bili_API_Service` 模块。
- **本地转码 (可选扩展):** **ffmpeg_kit_flutter**，用于处理导出为 MP3/FLAC 的需求。

## 3. 系统架构设计

为了保证软件的可维护性，特别是应对 B站 API 频繁变动的风险，系统采用**分层架构**与**解析模块解耦**设计。

1. **UI 视图层 (Presentation Layer):**
   - 包含侧边栏（导航）、主内容区（歌单/搜索/多P选择 GUI）、底部播放控制条。
   - 负责呈现精美的动效（如封面虚化背景、播放频谱等）。
2. **业务逻辑层 (State/Business Logic Layer):**
   - **播放控制器 (Player Controller):** 管理播放队列、播放模式（单曲循环、随机）、控制 `media_kit`。
   - **歌单管理器 (Playlist Manager):** 处理歌曲的增删改查。
   - **同步与账号模块 (Auth Service):** 处理二维码扫码登录、Cookie 保存与自动刷新。
3. **核心服务层 (Core Services):**
   - **Bili 解析引擎 (Bili Parsing Engine):** 这是**解耦的核心**。专门负责将 BV 号转化为真实的音频直链。包含 WBI 签名计算、多 P 列表获取。*未来如果 API 变更，只需更新此模块的逻辑即可。*
   - **缓存/下载管理器 (Download Manager):** 监听下载进度，将媒体流保存到本地特定目录，并将本地路径映射到数据库。
   - **元数据服务 (Metadata Service):** 处理默认信息（UP主=歌手，标题=歌名）的抓取，并接收用户的自定义覆盖 (Override) 数据。

## 4. 核心功能实现思路

### 4.1 B站账号登录与鉴权

- **思路:** 不建议使用账号密码（容易触发风控）。采用**扫码登录**方案。
- **实现:** 软件调用 B站生成登录二维码的 API，在 UI 展示。用户用手机 B站 App 扫码后，轮询接口获取登录状态，成功后将返回的 `SESSDATA`, `bili_jct` 等 Cookie 字段持久化存储。后续所有获取音频流的请求都携带这些 Cookie，以解锁高音质（如 192k、无损）。

### 4.2 链接解析与多 P 处理

- **思路:** 用户输入 BV 号 -> 引擎请求视频信息 -> 如果是多 P，弹出专用的选择 GUI。
- **实现:** 1. 调用 B站 `web-interface/view` 接口获取视频信息。 2. 提取 `pages` 数组。如果 `pages.length > 1`，则触发 UI 层的“多P解析弹窗”，展示每一 P 的标题。 3. 用户勾选需要的曲目后，生成带有独立 `CID`（B站的分P唯一标识）的内部歌曲对象。

### 4.3 播放与本地缓存机制

- **思路:** 流媒体播放与后台下载相结合。
- **实现:**
  1. 当用户点击播放时，解析引擎获取当前 `CID` 的真实 `m4s` 音频流地址。
  2. 播放器直接播放该网络 URL。
  3. 如果用户开启了“自动缓存”，后台下载模块同时将该流下载到本地目录。
  4. 数据库记录该歌曲的本地绝对路径。下次播放时，优先检查本地路径，存在则直接播放本地文件，做到“零缓冲”。

### 4.4 元数据自定义 (Metadata Override)

- **思路:** 保护原始数据，叠加用户数据。
- **实现:** 数据库设计时，歌曲表 (Song Table) 包含两组字段：`origin_title`, `origin_artist`（抓取自 B站）和 `custom_title`, `custom_artist`（用户填写）。UI 层展示时，优先读取 `custom` 字段，为空时 fallback 到 `origin` 字段。这样即便用户重置，也能找回原始视频信息。
